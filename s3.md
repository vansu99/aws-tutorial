# Amazon S3

Amazon S3 là một trong những dịch vụ nền tảng quan trọng nhất của AWS. S3 là dịch vụ lưu trữ object có khả năng mở rộng lớn, bảo mật tốt, hỗ trợ nhiều storage class, lifecycle, versioning, encryption, replication, access logs và rất phù hợp cho static files, website hosting, backup, media hosting.

---

## 1. Khái niệm và phép ẩn dụ

## Amazon S3 là gì?

**Amazon S3 — Simple Storage Service** là dịch vụ lưu trữ object của AWS. Nói đơn giản:

> S3 giống như một “kho lưu trữ file online” dành cho hệ thống backend.

Bạn có thể lưu:

- Ảnh user upload
- Video
- PDF
- File log
- File backup
- File tĩnh của website: HTML, CSS, JS
- Dataset lớn cho analytics

Ví dụ với ứng dụng Laravel:

```text
User upload ảnh avatar
        ↓
Laravel nhận file
        ↓
Laravel upload file lên S3
        ↓
Database chỉ lưu URL hoặc object key
        ↓
User truy cập ảnh qua CloudFront hoặc pre-signed URL
```

## Phép ẩn dụ dễ hiểu

Hãy tưởng tượng bạn có một **nhà kho lưu trữ hàng hóa**:

| Đời thực | Amazon S3 |
|---|---|
| Nhà kho | Bucket |
| Món hàng trong kho | Object |
| Mã vị trí / mã hàng | Key |
| Khu hàng thường dùng | S3 Standard |
| Khu hàng ít dùng | S3 Standard-IA |
| Kho lạnh / kho lưu trữ lâu năm | Glacier / Deep Archive |
| Quy định ai được vào kho | IAM Policy / Bucket Policy |
| Camera giám sát | Access Logs / CloudTrail |
| Tự động chuyển hàng cũ sang kho rẻ hơn | Lifecycle Rule |

## Vì sao gọi là “Simple Storage Service”?

Gọi là **Simple** vì người dùng không cần tự quản lý ổ cứng, RAID, server lưu file, replication hay capacity planning.

Bạn chỉ cần:

```text
Tạo bucket → Upload object → Phân quyền → Sử dụng
```

AWS lo phần còn lại như durability, scalability, availability, encryption mặc định, và khả năng lưu trữ cực lớn.

## S3 khác gì EBS?

Điểm rất quan trọng cho người mới học Cloud:

| Tiêu chí | Amazon S3 | Amazon EBS |
|---|---|---|
| Loại storage | Object Storage | Block Storage |
| Dùng để làm gì | Lưu file, ảnh, video, backup, static website | Gắn như ổ đĩa cho EC2 |
| Gắn trực tiếp vào EC2? | Không | Có |
| Truy cập bằng | HTTP API, SDK, CLI | File system của OS |
| Ví dụ | `s3://my-bucket/images/a.jpg` | `/var/www/html`, `/data/mysql` |
| Phù hợp | File upload, media, backup, log | OS, database, application data local |
| Scale | Gần như tự động | Theo dung lượng volume |
| Chia sẻ nhiều server | Dễ qua API | Khó hơn, thường 1 EC2 / volume |

**Kết luận dễ nhớ:**

```text
EBS = ổ cứng gắn vào server EC2
S3  = kho file online cho ứng dụng và hệ thống
```

# 2. Các thành phần then chốt của Amazon S3

## 2.1. Bucket

**Bucket** là nơi chứa dữ liệu trong S3.

Ví dụ:

```text
my-company-upload-bucket
video-app-prod-storage
onetech-backup-logs
```

Một số điểm cần nhớ:

- Bucket name phải **unique toàn cầu**.
- Bucket được tạo trong một **AWS Region**.
- Tên bucket nên dùng chữ thường, số, dấu gạch ngang.
- Không nên dùng tên quá chung chung như `test`, `upload`, `backup`.

Ví dụ đặt tên tốt:

```text
company-prod-user-uploads-apne1
company-stg-backup-apne1
```

---

## 2.2. Object

**Object** là file được lưu trong S3.

Một object thường gồm:

```text
File data
Metadata
Tags
Key
Version ID nếu bật versioning
```

Ví dụ object:

```text
avatar.png
invoice-2026-06.pdf
video-001.mp4
backup-20260614.sql.gz
```

Object size tối đa trong S3 là **5 TB**.

---

## 2.3. Key

**Key** là tên định danh đầy đủ của object trong bucket.

Ví dụ:

```text
uploads/users/1001/avatar.png
videos/2026/06/demo.mp4
backups/mysql/2026-06-14.sql.gz
```

Trong S3 thực tế không có folder thật như Linux. Thứ bạn thấy giống folder là do **prefix trong key**.

Ví dụ:

```text
s3://my-bucket/uploads/users/1001/avatar.png
```

Trong đó:

```text
Bucket: my-bucket
Key: uploads/users/1001/avatar.png
Prefix: uploads/users/1001/
Object name: avatar.png
```

---

## 2.4. Storage Classes

Storage Class quyết định **giá lưu trữ**, **tốc độ truy xuất**, **độ sẵn sàng**, và **phí retrieval**.

| Storage Class | Dùng khi nào | Ghi nhớ nhanh |
|---|---|---|
| S3 Standard | File truy cập thường xuyên | Nhanh, bền, phổ biến nhất |
| S3 Intelligent-Tiering | Không biết file sẽ được truy cập nhiều hay ít | AWS tự tối ưu tier |
| S3 Standard-IA | File ít truy cập nhưng cần lấy nhanh | Rẻ hơn Standard, có phí retrieval |
| S3 One Zone-IA | File ít truy cập, có thể tạo lại được | Rẻ hơn nhưng chỉ lưu 1 AZ |
| Glacier Instant Retrieval | Archive nhưng cần lấy ngay | Dữ liệu lạnh nhưng cần truy cập milliseconds |
| Glacier Flexible Retrieval | Archive, chấp nhận chờ vài phút đến vài giờ | Backup dài hạn |
| Glacier Deep Archive | Lưu rất lâu, rất ít truy cập | Rẻ nhất, restore lâu |

## Cách chọn nhanh

| Nhu cầu | Nên dùng |
|---|---|
| Ảnh/video user xem thường xuyên | S3 Standard |
| Không biết pattern truy cập | Intelligent-Tiering |
| Backup DB 30–90 ngày | Standard-IA hoặc Glacier Instant Retrieval |
| Log cũ ít đọc | Glacier Flexible Retrieval |
| Dữ liệu compliance giữ nhiều năm | Glacier Deep Archive |
| Thumbnail có thể generate lại | One Zone-IA hoặc xóa bằng Lifecycle |

---

## 2.5. Versioning

**Versioning** giúp lưu nhiều phiên bản của cùng một object.

Ví dụ bạn upload file cùng key:

```text
uploads/avatar.png
```

Lần 1:

```text
version-id: v1
```

Lần 2 overwrite:

```text
version-id: v2
```

Lần 3 overwrite:

```text
version-id: v3
```

Lợi ích:

- Khôi phục file bị overwrite nhầm.
- Khôi phục file bị delete nhầm.
- Bảo vệ dữ liệu quan trọng.
- Hữu ích cho backup, tài liệu, file hợp đồng.

Lưu ý:

```text
Versioning tăng chi phí vì mỗi version cũ cũng được tính storage.
```

Vì vậy nên kết hợp với **Lifecycle Rule** để xóa version cũ sau một thời gian.

---

## 2.6. Lifecycle Rules

**Lifecycle Rule** là cơ chế tự động chuyển hoặc xóa object để tối ưu chi phí.

Ví dụ:

```text
Ngày 0-30: S3 Standard
Sau 30 ngày: chuyển sang Standard-IA
Sau 90 ngày: chuyển sang Glacier
Sau 365 ngày: xóa
```

Lifecycle có 2 nhóm chính:

| Loại action | Ý nghĩa |
|---|---|
| Transition | Chuyển object sang storage class khác |
| Expiration | Xóa object sau một thời gian |

Ví dụ thực tế:

```text
uploads/images/*
- Sau 30 ngày chuyển sang Standard-IA
- Sau 180 ngày chuyển sang Glacier
```

```text
logs/*
- Sau 90 ngày chuyển sang Glacier
- Sau 365 ngày xóa
```

```text
tmp/*
- Sau 7 ngày xóa
```

Lifecycle có thể dùng để transition object sang class khác, xóa object sau một thời gian, xóa old versions, xóa incomplete multipart uploads, và có thể áp dụng theo prefix hoặc tag.

---

## 2.7. Bucket Policy và IAM Policy

S3 có 2 kiểu phân quyền rất hay gặp:

| Loại policy | Gắn ở đâu | Dùng khi nào |
|---|---|---|
| IAM Policy | User / Group / Role | Cấp quyền cho người dùng, EC2, Lambda |
| Bucket Policy | Bucket | Cấp rule trực tiếp trên bucket, public access, cross-account, force HTTPS/encryption |

## IAM Policy

Ví dụ: EC2 instance cần upload file lên S3.

Best practice là:

```text
EC2 → IAM Role → S3
```

Không nên:

```text
EC2 source code chứa AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY
```

## Bucket Policy

Bucket Policy thường dùng để:

- Cho phép public read khi host static website.
- Cho phép CloudFront truy cập S3.
- Chặn truy cập không dùng HTTPS.
- Force encryption khi upload.
- Cho phép cross-account access.

Một IAM principal truy cập được S3 object nếu IAM permission allow hoặc resource policy allow, đồng thời không có explicit deny.

---

## 2.8. Encryption

S3 hỗ trợ mã hóa dữ liệu để bảo vệ file.

| Loại encryption | Ý nghĩa | Khi nào dùng |
|---|---|---|
| SSE-S3 | AWS S3 tự quản lý key | Default, đơn giản, đủ cho nhiều workload |
| SSE-KMS | Dùng AWS KMS để quản lý key | Cần audit, control key, compliance |
| SSE-C | Customer tự cung cấp key | Ít dùng, phức tạp |
| Client-side encryption | App tự encrypt trước khi upload | Khi muốn tự kiểm soát hoàn toàn |

## SSE-S3

SSE-S3 là cách đơn giản nhất. AWS quản lý key, object được encrypt server-side.

## SSE-KMS

SSE-KMS dùng AWS KMS. Ưu điểm:

- Quản lý key tốt hơn.
- Audit key usage bằng CloudTrail.
- Có thể dùng customer managed key.
- Phù hợp dữ liệu nhạy cảm.

Ví dụ:

```text
Invoice
Contract
Personal data
Financial reports
```

---

## 2.9. Static Website Hosting

S3 có thể host website tĩnh:

```text
index.html
style.css
app.js
images/logo.png
```

Phù hợp cho:

- Landing page
- Documentation site
- Frontend build từ React/Vue/Angular
- Static marketing website

Nhưng cần nhớ:

```text
S3 static website hosting chỉ phục vụ static content.
Không chạy PHP, Laravel, Node.js, Python backend.
```

Muốn public website bằng S3 thì thường cần:

- Enable Static Website Hosting.
- Upload `index.html`.
- Tắt Block Public Access cho bucket đó.
- Thêm Bucket Policy cho phép public read.

Trong production, nên dùng:

```text
User → CloudFront → S3
```

Thay vì:

```text
User → S3 website endpoint
```

# 3. Ví dụ thực hành: Tạo S3 Bucket lưu ảnh upload từ Web App

## Kịch bản

Bạn có ứng dụng Web cho phép user upload avatar.

Kiến trúc đề xuất:

```text
User
 ↓
Web App Laravel trên EC2
 ↓
IAM Role attached to EC2
 ↓
Amazon S3 Bucket
 ↓
CloudFront phân phối ảnh cho user
```

## Mục tiêu

Tạo bucket để lưu:

```text
uploads/images/{user_id}/{filename}
```

Ví dụ:

```text
uploads/images/1001/avatar.png
uploads/images/1002/profile.jpg
```

---

## Bước 1: Tạo bucket

Thông tin đề xuất:

```text
Bucket name: myapp-prod-user-uploads-apne1
Region: ap-northeast-1
Object Ownership: Bucket owner enforced
Block Public Access: ON
Versioning: ON
Default encryption: SSE-S3 hoặc SSE-KMS
```

Với app upload file private, nên để:

```text
Block Public Access = ON
```

Không public bucket nếu không thật sự cần.

---

## Bước 2: Cấu hình Block Public Access

Với file upload user, nên giữ:

```text
Block all public access: ON
```

User không truy cập trực tiếp public object. Thay vào đó dùng:

```text
Pre-signed URL
CloudFront signed URL/cookie
Backend proxy
```

Với public asset như logo, CSS, JS có thể dùng CloudFront phía trước S3.

---

## Bước 3: Upload file

Có thể upload bằng AWS Console để test:

```text
S3 → Bucket → Upload → Add files → Upload
```

Key ví dụ:

```text
uploads/images/1001/avatar.png
```

---

## Bước 4: Phân quyền bằng IAM Role

Best practice:

```text
EC2 instance profile / IAM Role
```

Policy tối thiểu cho EC2 upload ảnh:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "UploadUserImages",
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:GetObject",
        "s3:DeleteObject"
      ],
      "Resource": "arn:aws:s3:::myapp-prod-user-uploads-apne1/uploads/images/*"
    },
    {
      "Sid": "ListBucketSpecificPrefix",
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::myapp-prod-user-uploads-apne1",
      "Condition": {
        "StringLike": {
          "s3:prefix": [
            "uploads/images/*"
          ]
        }
      }
    }
  ]
}
```

Ý nghĩa:

```text
Chỉ cho phép app thao tác trong prefix uploads/images/*
Không cho toàn quyền cả bucket nếu không cần.
```

---

## Bước 5: Bật Versioning

Nên bật Versioning cho dữ liệu quan trọng.

Ví dụ:

```text
User upload nhầm avatar mới
Admin có thể restore version cũ
```

Nhưng cần lifecycle để dọn version cũ, ví dụ:

```text
Noncurrent versions → xóa sau 30 ngày
```

---

## Bước 6: Bật Encryption

Khuyến nghị:

```text
Default encryption: SSE-S3
```

Nếu dữ liệu nhạy cảm:

```text
Default encryption: SSE-KMS
```

Ví dụ với SSE-KMS:

```text
S3 bucket dùng KMS key riêng
CloudTrail audit được ai dùng key
Có thể kiểm soát quyền decrypt/encrypt
```

---

## Bước 7: Tạo Lifecycle Rule

Ví dụ rule cho ảnh upload:

| Điều kiện | Action |
|---|---|
| Object prefix | `uploads/images/` |
| Sau 30 ngày | Chuyển sang Standard-IA |
| Sau 180 ngày | Chuyển sang Glacier Instant Retrieval hoặc Glacier Flexible Retrieval |
| Noncurrent version sau 30 ngày | Xóa |
| Incomplete multipart upload sau 7 ngày | Xóa |

Ví dụ logic:

```text
Ảnh mới thường được xem nhiều trong 30 ngày đầu.
Sau đó ít xem hơn → chuyển sang class rẻ hơn.
Version cũ không cần giữ mãi → xóa sau 30 ngày.
```

## AWS CLI thực hành

## Tạo bucket

Với Tokyo Region:

```bash
aws s3api create-bucket \
  --bucket myapp-prod-user-uploads-apne1 \
  --region ap-northeast-1 \
  --create-bucket-configuration LocationConstraint=ap-northeast-1
```

## Bật Block Public Access

```bash
aws s3api put-public-access-block \
  --bucket myapp-prod-user-uploads-apne1 \
  --public-access-block-configuration \
  BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
```

## Bật Versioning

```bash
aws s3api put-bucket-versioning \
  --bucket myapp-prod-user-uploads-apne1 \
  --versioning-configuration Status=Enabled
```

## Bật default encryption SSE-S3

```bash
aws s3api put-bucket-encryption \
  --bucket myapp-prod-user-uploads-apne1 \
  --server-side-encryption-configuration '{
    "Rules": [
      {
        "ApplyServerSideEncryptionByDefault": {
          "SSEAlgorithm": "AES256"
        }
      }
    ]
  }'
```

## Upload file lên S3

```bash
aws s3 cp ./avatar.png s3://myapp-prod-user-uploads-apne1/uploads/images/1001/avatar.png
```

## Upload kèm content type

```bash
aws s3 cp ./avatar.png s3://myapp-prod-user-uploads-apne1/uploads/images/1001/avatar.png \
  --content-type image/png
```

## Tạo pre-signed URL để xem file private

```bash
aws s3 presign s3://myapp-prod-user-uploads-apne1/uploads/images/1001/avatar.png \
  --expires-in 3600
```

## Ví dụ code PHP/Laravel upload S3

Cấu hình `.env`:

```env
FILESYSTEM_DISK=s3
AWS_DEFAULT_REGION=ap-northeast-1
AWS_BUCKET=myapp-prod-user-uploads-apne1
```

Nếu app chạy trên EC2 có IAM Role, **không cần hardcode access key** trong `.env`.

Code upload:

```php
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Storage;

public function uploadAvatar(Request $request)
{
    $request->validate([
        'avatar' => 'required|image|max:5120',
    ]);

    $userId = auth()->id();
    $file = $request->file('avatar');

    $path = $file->storeAs(
        "uploads/images/{$userId}",
        'avatar.' . $file->getClientOriginalExtension(),
        's3'
    );

    return response()->json([
        'message' => 'Upload successfully',
        's3_key' => $path,
    ]);
}
```

Tạo temporary URL:

```php
$url = Storage::disk('s3')->temporaryUrl(
    'uploads/images/1001/avatar.png',
    now()->addMinutes(30)
);
```

# 4. AWS Best Practices cho Amazon S3

## 1. Không public bucket nếu không thật sự cần

Mặc định nên để:

```text
Block Public Access = ON
```

Chỉ public khi bạn thật sự host static website hoặc public asset.

Với ảnh/video user upload, nên dùng:

```text
Private bucket + CloudFront + signed URL/cookie
```

---

## 2. Dùng IAM Role thay vì Access Key hardcode

Không nên để trong source code:

```env
AWS_ACCESS_KEY_ID=xxx
AWS_SECRET_ACCESS_KEY=yyy
```

Nên dùng:

```text
EC2 IAM Role
Lambda Execution Role
ECS Task Role
```

Lý do:

- Không lộ key trong Git.
- Không cần rotate key thủ công.
- Dễ quản lý quyền theo least privilege.
- An toàn hơn rất nhiều.

---

## 3. Bật Encryption mặc định

Nên bật default encryption cho bucket:

```text
SSE-S3 cho workload thông thường
SSE-KMS cho dữ liệu cần audit/compliance
```

Nếu cần bắt buộc upload phải có encryption header, có thể dùng Bucket Policy để deny request không mã hóa.

---

## 4. Bật Versioning cho dữ liệu quan trọng

Nên bật cho:

```text
Tài liệu khách hàng
File hợp đồng
Backup
Config quan trọng
Source data
```

Nhưng nhớ thêm Lifecycle để xóa version cũ, tránh chi phí tăng âm thầm.

---

## 5. Sử dụng Lifecycle Policy để tối ưu chi phí

Ví dụ:

```text
logs/        → Glacier sau 90 ngày, xóa sau 365 ngày
backups/     → Standard-IA sau 30 ngày, Glacier sau 90 ngày
tmp/         → xóa sau 7 ngày
thumbnails/  → xóa sau 60 ngày nếu có thể tạo lại
```

Lifecycle là một trong những cách tiết kiệm chi phí S3 hiệu quả nhất.

---

## 6. Dùng CloudFront phía trước S3 khi phục vụ file cho end user

Không nên để user tải trực tiếp từ S3 nếu traffic lớn.

Nên dùng:

```text
User → CloudFront → S3
```

Lợi ích:

- Cache tại edge location.
- Giảm tải request về S3.
- Tăng tốc độ tải file.
- Hỗ trợ HTTPS, custom domain.
- Có thể dùng signed URL/cookie.
- Có thể chặn direct access vào S3 bằng Origin Access Control.

---

## 7. Bật logging/audit

Nên dùng:

```text
CloudTrail data events cho S3
S3 Server Access Logs nếu cần log truy cập object
```

Lưu ý quan trọng:

```text
Không log vào chính bucket đang được monitor.
```

Nếu không, có thể tạo logging loop làm bucket tăng dung lượng liên tục.

---

## 8. Theo dõi chi phí bằng S3 Storage Lens và Cost Explorer

Dùng để trả lời các câu hỏi:

```text
Bucket nào tăng dung lượng nhanh nhất?
Prefix nào tốn nhiều storage nhất?
Có nhiều incomplete multipart upload không?
Có nhiều noncurrent versions không?
Storage class đang dùng có hợp lý không?
```

Kiến trúc khuyến nghị:

```text
Frontend / Mobile App

Backend API

        ↓
CloudFront hoặc pre-signed URL
```

Best practice:

- Bucket private.
- Upload qua backend hoặc pre-signed URL.
- Dùng IAM Role cho backend.
- Bật encryption.
- Dùng CloudFront nếu file được xem nhiều.
- Dùng lifecycle cho file cũ.

Ví dụ thực tế:

```text
Ứng dụng video giống TikTok:
- Video gốc lưu ở S3
- Video convert/resized lưu ở S3
- CloudFront cache video cho user Nhật
- Lifecycle chuyển video cũ sang class rẻ hơn
```

## Kịch bản 2: Backup database dump, log, file hệ thống

Ví dụ:

```text
mysqldump hằng ngày
application log
nginx access log
report export
backup config
```

Flow:

```text
EC2 / RDS export job
        ↓
Compress file
        ↓
Upload lên S3
        ↓
Lifecycle sang Glacier
        ↓
Xóa sau thời gian retention
```

Ví dụ lifecycle:

| Loại file | Rule |
|---|---|
| DB backup daily | Standard 7 ngày → Standard-IA 30 ngày → Glacier 90 ngày |
| App logs | Standard 30 ngày → Glacier 90 ngày → Delete 365 ngày |
| Temporary exports | Delete sau 7 ngày |

Best practice:

- Bật versioning nếu backup quan trọng.
- Dùng SSE-KMS nếu backup chứa dữ liệu nhạy cảm.
- Chặn public access tuyệt đối.
- Bật audit bằng CloudTrail.

- Cache global.
- Tốc độ tốt hơn.

- Dễ tích hợp WAF.

# Tổng kết nhanh

Amazon S3 nên được hiểu là:

```text
Một kho lưu trữ object/file serverless, bền, scale lớn, bảo mật tốt,
phù hợp cho upload file, backup, media, static website và data lake.
```

Checklist khi tạo S3 bucket production:

```text
[ ] Bucket name rõ ràng theo environment / region
[ ] Block Public Access bật mặc định
[ ] IAM Role thay vì Access Key hardcode
[ ] Least privilege policy
[ ] Default Encryption bật
[ ] Versioning bật cho dữ liệu quan trọng
[ ] Lifecycle Rule để tối ưu chi phí
[ ] CloudFront phía trước nếu phục vụ end user
[ ] CloudTrail / Access Logs để audit
[ ] Theo dõi bằng Storage Lens / Cost Explorer
```

Cách nhớ nhanh:

```text
S3 = nơi lưu file
Bucket = thùng chứa
Object = file
Key = đường dẫn file
Storage Class = loại kho theo mức độ truy cập
Lifecycle = tự động dọn/chuyển kho
Versioning = cứu file khi ghi đè/xóa nhầm
IAM/Bucket Policy = ai được làm gì
Encryption = bảo vệ dữ liệu
CloudFront = phân phối file nhanh và tiết kiệm hơn
```
